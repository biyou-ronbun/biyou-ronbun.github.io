# 美容論文ブログ

学術論文を根拠にした美容ブログの運営プロジェクト。
読者は20〜40代の美容好き一般女性。公開先は note / はてなブログ。

## このブログの唯一の武器

**「根拠がある」と「誠実である」の2つだけ。**

他の美容メディアより速く書くことも、派手に書くこともできません。勝てるのは、
一次情報（論文）に当たっていること、そして「ここまでは言えない」を正直に書くことだけです。
この2つを崩す変更は、どんなに短期的に数字が出そうでも入れないでください。

## チーム

| 担当 | 役割 | 呼ぶタイミング |
|---|---|---|
| `secretary` | ネタ帳・進捗管理、采配、数値記録、文体ルールの番人 | **まずここ。** 「今週何やる?」「記事を1本回して」「振り返って」 |
| `researcher` | 論文を探して論文カードを作る | テーマが決まったら最初に |
| `writer` | 論文カードを元に記事を書く | 論文カードができてから |
| `x-operator` | X の投稿下書きを作る | 記事ができてから |
| `benchmark` | 伸びているブログ・Xを研究し、勝ち方の型を playbook に貯める | 数字が伸び悩んだとき、新しい型を仕入れたいとき |

### 仕事の流れ

```
secretary（次にやることを決める）
    ↓
researcher → research/<slug>.md（論文カード）
    ↓
writer → articles/<slug>.md（記事）
    ↓
x-operator → x/<年-月>.md（X投稿の下書き）
    ↓
secretary（台帳を更新）
```

**この順番を飛ばさないこと。** writer は論文カードが無ければ書けません。カード無しで書かせると、
このブログの唯一の武器が壊れます。

### 采配について

サブエージェントは他のサブエージェントを直接呼べません。
secretary は**指示書**を返してくるので、それを読んだメインのClaude（私）が実際に各担当を招集します。
オーナーから見れば「秘書に頼めば回る」で構いません。

## ディレクトリ

| パス | 中身 |
|---|---|
| `research/` | 論文カード。テーマ1つにつき1ファイル |
| `articles/` | ブログ記事（note / はてな向けの Markdown） |
| `x/` | X の投稿下書き。月ごとのカレンダー形式 |
| `ops/ledger.md` | ネタ帳・進捗表 |
| `ops/metrics.md` | PV・インプレッションの記録（**人間が管理画面から持ってきた数値**） |
| `ops/auto-metrics.md` | PV・収益の記録（**機械が公式APIから取った数値**。`auto/collect-metrics.mjs` が書く。手で編集しない） |
| `ops/retrospective.md` | 週次振り返り |
| `docs/style-guide.md` | 文体ルール・薬機法NG表現集 |
| `site/` | 自前サイト。`articles/*.md` から静的サイトを生成する（`site/README.md` 参照） |
| `site/books.json` | Kindle の巻の台帳。**本の設定はここ1箇所だけ**（EPUB もサイトの本紹介も、ここを読む） |
| `auto/` | 自動運転。記事の生成・ネタの補充・次の巻・X投稿・数値の取得（`auto/README.md` 参照） |
| `output/blog-articles.txt` | 全記事を1本にまとめたテキスト（note / はてなへの貼り付け用） |

スラッグは論文カード・記事で揃えること（`research/retinol-wrinkle.md` ↔ `articles/retinol-wrinkle.md`）。

## 自前サイト（GitHub Pages に自動公開）

記事を書いたら `site/articles.json` に1ブロック足して push するだけ。
GitHub Actions が `site/build.mjs` を回して、サイトを自動で公開する。

```powershell
git add -A; git commit -m "記事を追加"; git push
```

- ビルドの本体は `site/build.mjs`（Node、外部ライブラリなし）。`site/build.ps1` はそれを呼ぶだけのラッパー。
  **ビルドのロジックを2箇所に書かないこと。**
- 出力先は `site/dist/`。**毎回まるごと作り直されるので、ここを直接編集しない**（git 管理外）。
- `site/articles.json` の `published: false` で記事を寝かせられる（週2本ずつ出す運用のため）。
- **リポジトリは public。** `ops/`、`x/`、`output/` は `.gitignore` で公開対象から外してある。
  ここに秘密を書き足すときは、公開対象に入っていないか必ず確認すること。
- `.gitignore` は行末コメントを解釈しない。`ops/  # メモ` と書くと除外が効かなくなる。
- 手元で確認するだけなら `node site/build.mjs` → `site/dist/index.html` を開く。

## 全員が守るルール

1. **論文を捏造しない。** タイトル、著者、年、DOI、PMID、数値は、実際に開いたページに
   書いてあったものだけを書く。記憶から補完しない。見つからなければ「見つからない」と書く。
2. **論文カードにないことは記事に書かない。**
3. **薬機法に触れない。** `docs/style-guide.md` に従う。迷う表現は使わずに済ませる。
4. **数値（PV、収益、imp）は、実測値だけを記録する。推測で埋めない。**
   実測値と認めるのは次の2つだけ。**どちらなのかが分かる形で書くこと。**
   - **人間が管理画面から持ってきた数値**（`ops/metrics.md`）
   - **公式APIから機械が取得した数値**（`ops/auto-metrics.md`。`auto/collect-metrics.mjs` が書く）

   **取得に失敗したときは「取得失敗」と書く。0 と書かない。**
   「0だった」と「数えられなかった」は、別の事実です。**エージェントがこの2つのファイルに手で数字を書き足さないこと。**
5. **X への投稿は人間が手で行う。** エージェントは下書きを作るだけ。「投稿しました」と言わない。
6. **金の出口を、勝手に開けない。** 広告枠を出す（`showAdUnits`）、アフィリエイトのURLを入れる、
   本の `amazonUrl` を入れる——これらは**オーナーだけが行う**。エージェントは器を作るところまで。
