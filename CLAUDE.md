# 美容論文ブログ

学術論文を根拠にした美容ブログの運営プロジェクト。
読者は20〜40代の美容好き一般女性。公開先は note / はてなブログ。

## このブログの武器

**「論文を分析して伝える」こと。**

一次情報（論文）に当たり、数字はそのまま出し、**「ここまでは言えない」も正直に書く。**
他の美容メディアより速く書くことも、派手に書くこともできません。勝てるのはここだけです。

## ★★ 2026-07-14、方針が変わりました（オーナー判断）

**以前の憲法は「4つを断り続ける」ことを武器だと書いていました。**
（タイアップを断る / 商品を推薦しない / 判定を持たない / 検証した業界の広告を遮断する）

**オーナーの判断で、その方針は終わりました。**

> **「断り続けるのは将来的に利益が減ると思う」**
> **「ランキング・成分辞典・タイアップ・肌診断を開けなければ、読者がファン化しない」**

**うちの実測も、それを支持していました。**
`benchmark/general-beauty-media.md`: **占いは4社が持つ最強の再訪装置。** ランキングは @cosme / LIPS の中核。

**Consumer Reports は1936年から広告を断り続けた消費者団体。**

> **理念の強度では止まらない。86年の実績でも止まらなかった。**

**うちは、判定を持つことを選びました。**
**だから、いつか判定を売りたくなります。それは意志では止まりません。**

**★ そのときが来たら、この段落を読み返してください。**
**そして「判定を売った」なら、読者にそう書いてください。隠すことだけは、しないこと。**

### ★ では、匿名のまま、何でファンを作るのか

**うちが持てる「毎週、本当に変わるもの」は、いまのところ ランキング だけです。**

## チーム

| 担当 | 役割 | 呼ぶタイミング |
|---|---|---|
| `secretary` | ネタ帳・進捗管理、采配、数値記録、文体ルールの番人 | **まずここ。** 「今週何やる?」「記事を1本回して」「振り返って」 |
| `researcher` | 論文を探して論文カードを作る | テーマが決まったら最初に |
| `writer` | 論文カードを元に記事を書く | 論文カードができてから |
| `x-operator` | X の投稿下書きを作る | 記事ができてから |
| `benchmark` | 伸びているブログ・Xを研究し、勝ち方の型を playbook に貯める | 数字が伸び悩んだとき、新しい型を仕入れたいとき |

### 仕事の流れ

```
secretary（次にやることを決める）
    ↓
researcher → research/<slug>.md（論文カード）
    ↓
writer → articles/<slug>.md（記事）
    ↓
x-operator → x/<年-月>.md（X投稿の下書き）
    ↓
secretary（台帳を更新）
```

**この順番を飛ばさないこと。** writer は論文カードが無ければ書けません。カード無しで書かせると、
このブログの唯一の武器が壊れます。

### 采配について

サブエージェントは他のサブエージェントを直接呼べません。
secretary は**指示書**を返してくるので、それを読んだメインのClaude（私）が実際に各担当を招集します。
オーナーから見れば「秘書に頼めば回る」で構いません。

## 自前サイト（GitHub Pages に自動公開）

記事を書いたら `site/articles.json` に1ブロック足して push するだけ。
GitHub Actions が `site/build.mjs` を回して、サイトを自動で公開する。

```powershell
git add -A; git commit -m "記事を追加"; git push
```

- ビルドの本体は `site/build.mjs`（Node、外部ライブラリなし）。`site/build.ps1` はそれを呼ぶだけのラッパー。
  **ビルドのロジックを2箇所に書かないこと。**
- 出力先は `site/dist/`。**毎回まるごと作り直されるので、ここを直接編集しない**（git 管理外）。
- `site/articles.json` の `published: false` で記事を寝かせられる（週2本ずつ出す運用のため）。
- **リポジトリは public。** `ops/`、`x/`、`output/` は `.gitignore` で公開対象から外してある。
  ここに秘密を書き足すときは、公開対象に入っていないか必ず確認すること。
- `.gitignore` は行末コメントを解釈しない。`ops/  # メモ` と書くと除外が効かなくなる。
- 手元で確認するだけなら `node site/build.mjs` → `site/dist/index.html` を開く。

## 悩みタグ（読者の言葉で、記事への入口を作る）

カテゴリ（塗る／飲む／習慣）は**対象の分類**で、読者の悩みではない。読者は「塗るもの」を
探して来ない。「毛穴」「乾燥」で来る。だから `site/articles.json` の `tags` で入口を増やす。

**ここには、越えたらサイトが別物になる線が2本ある。**

1. **タグページに、記事へのリンク以外を1文字も書かない。**
   解説を書き始めた瞬間、それは「成分辞典」になる（企画として却下済み）。
2. **タグは読者の言葉だけ。「こちらの判定」をタグにしない。**
   「根拠なし」「動物実験」「メーカー資金」をタグにした瞬間、それは記事への入口ではなく
   **糾弾リストの目次**になる（却下済み）。

語彙は `site/build.mjs` の `TAG_VOCAB` に固定してある。それ以外を書くと
`site/verify.mjs` が公開を止める。語を増やしたくなったら、勝手に足さずオーナーに聞くこと。

## 配信の購読者を増やす（毎週木曜）— ★ コンテンツは作らない

```
木曜 8:00   auto/retention.ps1
  ↓
① 越えていないか見る（auto/retention-gates.mjs）
  ↓
② 購読者を測る（X / note / はてな。auto/push-metrics.mjs）
  ↓
③ 配信の導線を**1つだけ**直す（★ コンテンツは作らない）
  ↓
④ **★ 線を越えていたら、git で元に戻す**
```

### ★★ 「リピーターを増やすコンテンツ」は、作れない

**商品を勧めない媒体を4つ、実際に開いて調べた結果:**

> **再訪の理由が、コンテンツの中にあった例はゼロ。**
> 観測できたのは ①**配信（push）** ②**読者自身の次の疑問** の2つだけ。
> **「読者が喜ぶページを1枚足す」は、再訪の打ち手ではない。**

### ★ そして、うちは再訪を測れない

解析（Cloudflare）は **Cookie を使わない**（同意バナーを出さないため、そう選んだ）。
**Cookie が無ければ、「同じ人が戻ってきた」ことは原理的に分からない。**

**測れない数字を改善目標にすると、機械は必ず数字を捏造する。**

**だから目標は「再訪率」ではなく「配信の購読者数」。これは測れる。**
**測れるものと、効くものが、一致している。**

### ★★ この輪が、いちばん暴走しやすい

「リピーターを増やすコンテンツを毎週作れ」と命じられた機械は、必ずこうする:

| やること | なぜ駄目か |
|---|---|
| **2個目の診断・クイズ** | 再訪の分かれ目は「入力が来るたび変わるか」の1点。**うちの診断は入力が変わらない。一度で終わる。2個目も同じ** |
| **ポイント・バッジ・レベル・連続記録** | **読者を採点する装置。採点した瞬間、「騙されてましたね」に変わる** |
| **通知をねだる** | **通知は、追跡の入口** |
| **「毎日更新中!」** | **「また来てください」と言って人は来ない** |
| **「1,200人が読者」** | **購読者は 0 人。書いた瞬間、捏造** |

**既存の `/check.html` が採点していないのは正解。偶然にしないこと。**

### ★ 「今週は何もしませんでした」が、正しい仕事

**購読者が増えない最大の理由は、導線ではない。読者が数十人だから。**
**何かをしなければと思った瞬間、機械は2個目の診断を作り始める。**

### ★ 自動広告の「ページ内広告」は、OFF のまま（2026-07-13・オーナー判断）

AdSense の自動広告は **ON**。ただし **ページ内広告（インライン）だけ OFF**。

- ONにして増えるのは **月30〜70円**（PV 300/月・RPM 300〜800円で試算）
- **メンバーが1人つけば月980円。ページ内広告の14〜30倍**

**「収益を優先する」なら、Stripe を先にやるべき。** ページ内広告は、優先順位がいちばん低い。

**次に「収益を改善しろ」と言われた機械が、ここを戻したくなる。戻さないこと。**

## 海外の学術誌の新着を見る（毎週金曜）— ★ 翻訳も要約も記事化もしない

```
金曜 7:00   auto/journal.ps1
  ↓
① 海外10誌の新着を PubMed から拾う（auto/journal-watch.mjs）
  ↓
② ★ RCT・メタ解析・撤回が1本も無ければ、Claude を呼ばずに終わる
  ↓
③ あれば、3つに仕分ける
     ・既存記事に影響する → オーナーに報告（★ 記事は書き換えない）
     ・新しいネタ         → topics.json に queued で足す
     ・どちらでもない      → 何もしない
  ↓
④ ★ 記事を書いていたら、git で元に戻す
```

### ★★ 「新着 → 翻訳 → 要約 → 記事化」は、作れない

**それは「他人の情報を、右から左に流す」こと。**

**Google のスパムポリシーは「AIで価値を加えないページの大量生成」を明示的に禁じている**（自動か人力かを問わない）。
**このサイトを殺すのは、書かないことではなく、薄いものを書くこと。**

**化粧品メーカーの新着は、プレスリリース。** 翻訳して記事にすれば、**うちは無料の宣伝媒体**になる。

**うちがやっているのは「要約」ではない。「一次情報まで辿って、ここまでは言えませんと書くこと」。**

記事は `researcher → 論文カード → writer` で作る。
**カード無しで書かせると、このブログの唯一の武器が壊れる。**

**`auto/journal.ps1` は、この輪が `articles/` を触ったら git で元に戻す。**

### ★★ MeSH で「ヒトの研究だけ」に絞ると、新着が消える

**MeSH は、PubMed の司書が後から付ける索引語。索引付けは数週間〜数か月遅れる。**

実測（2026-07-14、過去30日）:

```
humans[mh] あり →  1 件
humans[mh] なし → 38 件   ← 37件を、索引の遅れで見落としていた
```

### ★★ なぜ「呼ばずに終わる」のか

**データが無いときに「需要を分析しろ」と言われた機械は、必ず推測でデータを作る。**
そして推測に基づいて記事を量産し始める。

**それは Google が最も嫌う「AI で価値を加えないページの大量生成」そのもの。**
**AdSense と検索流入を、同時に失う。**

**呼ばなければ、捏造は起きない。**

Search Console のデータは、登録から数週間かかる。
それまで毎週「まだデータがない」と記録して終わる。**それが正しい動作。**

**「0件」は「人気が無い」ではなく「データが存在しない」。この2つを混同しないこと。**

### ★ 直すのは、1週間に1つだけ

たくさん直すと、**何が効いたのか分からなくなる。**
そして「たくさん直した」こと自体が目的になる。

### ★ 需要を見るのは、読者の言葉を知るため。読者に迎合するためではない

**「この言葉で検索されているから、この結論を書こう」と考えたら、そこで終わり。**

## X の自動運転（研究 → 補充 → 投稿）

```
日曜 7:00    Xの型を研究   auto/x-research.ps1  → benchmark/x-playbook.md を更新
毎朝 6:00    Xの弾を補充   auto/x.ps1           → playbook を読んで投稿を作る（在庫30件未満のときだけ）
毎時 7〜23   Xへ投稿      auto/post-x.ps1      → auto/verify-x.mjs を通ったものだけ
```

**★ この輪には、構造的な危険がある。**

「伸びている投稿」を毎週取り込み続けると、**必ず煽りに寄る。** バズる投稿は、断定と煽りと不安で
できているからだ。毎週それを学習すれば、**このアカウントは数か月で、うちが批判している側と同じものになる。**

歯止めは2つ。**どちらかを外したくなったら、それが危険信号。**

1. `auto/x-research-prompt.md` が「**煽らずに伸びている型だけを持ち帰れ。それ以外は明示的に捨てて、
   捨てたことを記録しろ**」と命じている。「今週もこの型は持ち帰らなかった」と書き続けることが、
   あの仕事の本体
2. **`auto/verify-x.mjs` が、投稿の直前に全件を検査する。** 研究がどんな型を持ち帰っても、
   煽り・言い過ぎ・薬機法に触れる投稿は機械が拒否する。**1本でも落ちたら、その回は1本も出ない**

**関門のほうが、playbook より上位。**
playbook が「この煽り方が伸びる」と書いていたら、**間違っているのは playbook のほう。**

## 全員が守るルール

1. **論文を捏造しない。** タイトル、著者、年、DOI、PMID、数値は、実際に開いたページに
   書いてあったものだけを書く。記憶から補完しない。見つからなければ「見つからない」と書く。
2. **論文カードにないことは記事に書かない。**
3. **薬機法に触れない。** `docs/style-guide.md` に従う。迷う表現は使わずに済ませる。
4. **数値（PV、収益、imp）は、実測値だけを記録する。推測で埋めない。**
   実測値と認めるのは次の2つだけ。**どちらなのかが分かる形で書くこと。**
   - **人間が管理画面から持ってきた数値**（`ops/metrics.md`）
   - **公式APIから機械が取得した数値**（`ops/auto-metrics.md`。`auto/collect-metrics.mjs` が書く）

   **取得に失敗したときは「取得失敗」と書く。0 と書かない。**
   「0だった」と「数えられなかった」は、別の事実です。**エージェントがこの2つのファイルに手で数字を書き足さないこと。**
5. **X への投稿は人間が手で行う。** エージェントは下書きを作るだけ。「投稿しました」と言わない。
6. **金の出口を、勝手に開けない。** 広告枠を出す（`showAdUnits`）、アフィリエイトのURLを入れる、
   本の `amazonUrl` を入れる——これらは**オーナーだけが行う**。エージェントは器を作るところまで。
7. **リンクを付けられる記事が出たら、毎回オーナーに知らせる。**
   6と対になるルール。**開けられるのはオーナーだけなのだから、知らせなければ、その記事は永久に収益ゼロ。**
   記事は平日毎日1本増えるので、**知らせを人の記憶に頼れば必ず漏れる。**
   `auto/product-candidates.mjs` が一覧を作り、`ops/product-candidates.md` に書く。
   記事の輪（`auto/run.ps1`）と収益の輪（`auto/money.ps1`）の両方から、毎回呼ばれる。

   **ただし「基準が導けない記事」に、無理に商品を置かないこと。** 10本のうち5本がそれだった。
   **置かないことも、正しい結果。**
