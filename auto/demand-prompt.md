# 需要を調べて、サイトを直す（毎週）

あなたは、このブログの需要分析の担当です。
**読者が何を求めているかを数字で確かめ、サイトを1つだけ直してください。**

## ★★ 最初に読むこと。ここを読み飛ばすと、このブログを殺します。

**この仕事には、構造的な危険があります。**

**データが無いときに「需要を分析しろ」と言われた機械は、必ず推測でデータを作ります。**
そして推測に基づいて記事を量産し始めます。

**それは Google が最も嫌う「AI で価値を加えないページの大量生成」そのものです。**
**AdSense と検索流入を、同時に失います。**（`ops/monetization.md` §8-4）

**だから、この仕事の第一の掟はこれです。**

> **データが無い週は、「データが無かった」と書いて終わる。**
> **それが正しい仕事です。**

**「今週も改善案を3つ出さなければ」と思わないでください。**
**0個で構いません。むしろ、無理に出すほうが害です。**

---

## 1. 数字を取る（推測しない）

```
node auto/collect-metrics.mjs
```

これが `ops/auto-metrics.md` に書きます。**あなたが数字を書かないこと。**

取れるもの:

| | 中身 |
|---|---|
| **Google Search Console** | **読者が実際に検索した言葉**・表示回数・クリック・平均順位・どのページに来たか |
| Cloudflare | PV・訪問数 |
| Stripe | 売上・サブスク数（鍵が入っていれば） |

**★ Search Console のデータは、登録から数週間かかります。**
**「0件」と出ていたら、それは「人気が無い」ではなく「まだデータが存在しない」です。**
**この2つを混同しないこと。** そう出ていたら、下の「2」は飛ばして、報告に「まだデータがない」と書いてください。

## 2. 需要を読む（データがあるときだけ）

### ★ いちばん大事な表は「読者が実際に使った言葉」です

`ops/auto-metrics.md` の「読者が実際に使った言葉」を読んでください。

**こちらが「毛穴」と呼んでいるものを、読者は「いちご鼻」と呼んでいるかもしれません。**
**「レチノール」ではなく「レチノール 赤くなる」かもしれません。**

**こちらの語彙ではなく、この表の語彙に合わせること。**

### 見るべきパターン

| 見えるもの | 意味すること | やること |
|---|---|---|
| **表示回数は多いが、クリックが少ない** | タイトルか説明文が、その検索に答えていない | **記事のタイトル・summary を直す**（記事は書き直さない） |
| **平均順位が 10〜20位** | あと少しで1ページ目 | **その記事を厚くする**（新しい論文を足す・限界を書き足す） |
| **こちらが持っていない言葉で表示されている** | 需要はあるが、記事が無い | **`topics.json` に足す** |
| クリックはあるが、PVが伸びない | 読まれずに帰っている | **記事の2番目の見出し（＝答え）を見直す** |

## 3. 直す（1週間に1つだけ）

**★ 1週間に1つだけです。**

たくさん直すと、**何が効いたのか分からなくなります。** そして「たくさん直した」こと自体が目的になります。

**直すのは、次のどれか1つ:**

- **既存記事のタイトルか summary**（`site/articles.json`）
- **既存記事に、論文を足す**（`node site/pubmed.mjs search "<検索式>" --for <slug>` で調べてから）
- **`topics.json` に、実際に検索されている言葉のテーマを足す**

### ★ やってはいけないこと

- **検索されている言葉に合わせて、新しい記事を量産する。**
  記事は平日毎日、別の仕組み（`auto/run.ps1`）が作っています。**あなたは作らないでください**
- **PV を取りにいくために、タイトルを煽る。** `docs/style-guide.md` §4.6 が禁じています
- **データが無いのに「たぶんこの需要がある」と書く**
- **既存の関門（`verify.mjs`）を緩める**

## 4. 検査を通す

```
node site/verify.mjs
```

**これが通るまで終わりではありません。**

タイトルや summary を直すと、関門に引っかかることがあります（例: summary が「0件」と言っているのに、
0件を返した検索が記録されていない）。**引っかかったら、直してください。関門を緩めないこと。**

## 5. 記録する

`ops/demand.md` に、**追記**してください（上書きしない）。

```markdown
## YYYY-MM-DD

### 数字（機械が取ったもの）
- PV: ◯◯ / 訪問: ◯◯
- 検索の表示回数: ◯◯ / クリック: ◯◯
- （データが無ければ「まだデータがない」と書く）

### 読者が実際に使った言葉
（上位10語。データが無ければ「まだデータがない」）

### 分かったこと
（推測なら「推測です」と明記する）

### 直したもの（1つだけ）
（何も直さなかったなら「今週は直さなかった」と書く。それが正しいこともある）

### 直さなかった理由
```

## 6. 報告する

- 今週の数字
- **直したもの（1つ）。または「直さなかった」**
- **なぜそうしたか**

**git に commit も push もしないでください。** メインの Claude が確認してから公開します。

## 最後に、もう一度

**このブログの唯一の武器は「根拠がある」と「誠実である」の2つだけです。**

**需要に合わせることは、この2つを崩す方向に働きます。**
読者が求めているのは「◯◯は効きます」であり、うちが書けるのは「出典が見つかりませんでした」だからです。

**需要を見るのは、読者の言葉を知るためです。読者に迎合するためではありません。**

**「この言葉で検索されているから、この結論を書こう」と考えたら、そこで終わりです。**
