あなたは美容論文ブログ（https://biyou-ronbun.com/）の**ネタ帳を補充**します。人間は見ていません。

記事の自動生成は `topics.json` の待ち行列から1本ずつ取っていきます。**この行列が空になると、記事の自動生成が止まります。**
いま行列が残り少なくなったので、あなたが補充します。

## 最初に読むもの

- `CLAUDE.md` — このブログの憲法
- `docs/style-guide.md` — 文体と薬機法
- `topics.json` — ネタの待ち行列（ここに足す）
- `site/articles.json` — すでに公開した記事の一覧（**重複を避けるため**）
- `ops/ledger.md` — ネタ帳の本体（あれば。読者から届いた質問が入っていることがある）
- `ops/metrics.md` — 届いた質問のログ（あれば）

## このブログの勝ちパターン

> **「世間の常識の出典をたどると、根拠が無い / 動物実験だった / 試験管の中だけだった / メーカーが資金を出していた」**

この構造が再現できるテーマだけを入れてください。
**「〇〇の効果的な使い方」のような、ただの解説記事になるテーマは入れないこと。** それはPV目当ての薄い記事であり、
`docs/style-guide.md` §4.6 と `ops/monetization.md` §8-4（Google の scaled content abuse ポリシー）に反します。

## 1. 候補を出す

**8件**の候補を出してください。次の3つの井戸から汲みます。

1. **読者から届いた質問**（`ops/metrics.md` の「届いた質問のログ」）。**これが最優先です。**
   読者が実際に疑問に思ったことは、検索する人がいるということです。
2. **既存記事からの連想。** すでに書いた記事と相互リンクできるテーマ（例: `sunscreen-spf` → 日焼け後のケア）。
3. **美容の定番の言い回し。** 広告やSNSで繰り返し見るのに、出典が示されないもの。

**すでに `site/articles.json` にある記事、すでに `topics.json` にある slug は、絶対に重複させないこと。**

## 2. 【関門】書けるテーマかを、PubMed で確かめる

**ここを飛ばさないでください。** 論文が1本も無いテーマを行列に入れると、
その番が回ってきたときに記事が書けず、**自動生成が止まります。**

候補1件ずつについて、必ず実行する:

```
node site/pubmed.mjs search "<英語の検索語>"
```

**判定:**

- **ヒト臨床試験・システマティックレビューが1本以上ある** → 採用。「根拠がある」型でも「根拠が弱い」型でも記事になる
- **動物実験・in vitro しか無い** → **積極的に採用。** これはこのブログの一番強い型です（「出典は動物実験だった」）
- **論文が1本も見つからない** → **これも採用してよい。** ただし `angle` に
  **「PubMed で検索してもヒットしないことを、記事の結論にする」**と明記すること。
  `turnover-28days`（0件だったことが記事になった）が前例です
- **検索語が悪くて見つからないだけ、の可能性がある** → 検索語を2〜3通り変えて試す。それでも駄目なら上の扱い

**⚠️ 論文を捏造しないこと。** ここで見た検索結果に無い論文を、`angle` に書かないでください。
PMID や数値を `topics.json` に書く必要はありません。**書くなら、実際に見たものだけ。**

## 3. topics.json に足す

`topics` 配列の**末尾に**、採用したものを足してください。既存の要素は1つも変更しないこと。

```json
{
  "slug": "英数字とハイフンだけ。既存と重複しないこと",
  "theme": "記事の問い。「〜は本当か」「〜の出典はどこにあるのか」の形",
  "angle": "どういう構造で記事になるかの見込み。PubMedで実際に見た手応えを書く。検索して0件だったなら、そう書く",
  "status": "queued"
}
```

**目標: 補充後、`status: "queued"` が 8件以上あること。**

## 4. 確認して、push する

```
node -e "const t=require('./topics.json'); const q=t.topics.filter(x=>x.status==='queued'); console.log('queued:', q.length); const s=t.topics.map(x=>x.slug); if(new Set(s).size!==s.length) { console.error('!! slug が重複しています'); process.exit(1); } console.log('slug の重複なし');"
```

これが通ったら:

```
git add topics.json
git commit -m "ネタを補充"
git push origin main
```

**JSON が壊れていたら push しないこと。** 壊れた `topics.json` は、以降の自動生成を全部止めます。

## 5. 報告する

最後に、次の形で報告してください。

```
補充: N件
  - <slug>: <theme>   （PubMed: ヒト試験N本 / 動物実験のみ / 0件）
  ...
queued の残り: N件
```

**PubMed で0件だったテーマは、報告に必ず「0件」と書いてください。** それは失敗ではなく、記事の結論になります。
