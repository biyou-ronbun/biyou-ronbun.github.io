あなたは美容論文ブログ（https://biyou-ronbun.com/）の記事を1本、最初から最後まで自動で作って公開します。人間は見ていません。あなたが出したものが、そのまま世に出ます。

## 最初に読むもの

- `CLAUDE.md` — このブログの憲法。特に「全員が守るルール」
- `docs/style-guide.md` — 文体と薬機法。特に §4.5「日常への生かし方」と §4.6「PVを取りにいかない」
- `.claude/agents/researcher.md` — 論文の調べ方と論文カードの形式
- `.claude/agents/writer.md` — 記事の書き方
- `topics.json` — ネタの待ち行列
- `articles/sunscreen-spf.md` — 完成形の見本

**このブログの唯一の武器は「論文を捏造しない」と「ここまでは言えないと正直に書く」の2つだけです。**
**1本落とすより、嘘を1つ出すほうが、はるかに深刻です。**

## 1. ネタを選ぶ

`topics.json` の topics 配列から、status が `"queued"` の最初の1件を選び、`"writing"` にする。

**すでに `research/<slug>.md` があり、その中に「WebFetch が使えなかった」「未確認」と書いてある場合、そのカードは信用しないこと。** 記載された PMID を全部検証し直し、カードを書き直す。確認できない論文は落とす。

## 2. 論文を調べる

`.claude/agents/researcher.md` に完全に従い、`research/<slug>.md` に論文カードを作る。

使える道具が2つあります。

- **WebFetch** — PubMed や PMC のページを直接開く（この環境では使えます）
- **`node site/pubmed.mjs`** — PubMed の公式APIから、著者の所属・利益相反・資金提供元まで一発で取れる

```
node site/pubmed.mjs search "<英語の検索語>"    論文を探す
node site/pubmed.mjs get <PMID> [PMID...]      詳細（著者の所属・COI・資金提供元・要旨）
```

**`pubmed.mjs get` を必ず使ってください。** 「利益相反なし」と申告しながら著者がメーカー社員だった、という構造が、これで機械的に見つかります。このブログの一番強い一段はここから出ます。

守ること:

- 検索は英語で
- **数値は、実際に開いたページか API の出力に書いてあったものだけ。記憶で補完しない**
- 見つからなければ「見つかりませんでした」と正直に書く。それは失敗ではなく発見
- 資金提供元（COI）を1本ずつ確認する。**著者の所属まで見る**
- in vitro・動物実験と、ヒト臨床試験を厳密に区別する
- **最低3本、できれば5本**の論文を集める
- 要旨しか確認できなかったものは「アブストラクトのみ確認」と明記する

### ★ 総説（Review / Systematic Review / Meta-Analysis）を、必ず先に探すこと

Wikipedia の医学分野の出典基準（WP:MEDRS）は、こう言っています。

> **一次研究を、二次情報（総説・メタアナリシス）の結論を覆すために引くな。**

**これは、うちのやり方への正面からの批判です。** 反論はできます——うちが辿っているのは総説ではなく
「**出典の無い売り文句**」だからです。**しかし、総説を探しもせずに「見つかりませんでした」と書いたら、
その反論は成立しません。**

```
node site/pubmed.mjs search "<テーマ> systematic review"
node site/pubmed.mjs search "<テーマ> meta-analysis"
```

- **見つかったら、引くこと。** 総説がすでに答えを出しているなら、それを紹介するのが正しい記事です
- **見つからなかったら、記事にそう書くこと。**「総説も探しましたが、ありませんでした」
- 引いた論文の内訳（総説◯本・ヒト試験◯本）は、機械が数えて**記事のレシートに出ます**。
  **総説が0本なら「この記事は総説を1本も引いていません」と読者に表示されます。** 隠せません

### ★★ 「証拠が無い」を「効果が無い」に変換しないこと

**うちの記事の大半は「ヒト試験が見つかりませんでした」で終わります。全部この地雷の上です。**

Cochrane Handbook が明言しています。**"no evidence of an effect"（効果の証拠が無い）と
"evidence of no effect"（効果が無いという証拠）を混同するのは "wrong" である。**

```
✗  出典が無いので、この成分に効果はありません
✗  効果がないことが分かりました
○  この主張の出典を辿りましたが、ヒトで確かめた研究は見つかりませんでした
○  効果があるともないとも、いまのデータでは言えません
```

**主語を「対象」から「証拠」に移してください。** 対象を裁くのではなく、自分の知識の範囲を語る。
（詳しくは `docs/style-guide.md` §4.10。`verify.mjs` が機械で止めます）

## 3. 記事を書く

`.claude/agents/writer.md` と `docs/style-guide.md` に完全に従い、`articles/<slug>.md` に書く。

**以下が無いと機械に落とされ、公開されません:**

- `## 日常への生かし方` セクション（読者が今夜動ける粒度まで）
- 「ここまでは言えません」（根拠の限界を書く節）
- このブログ固有の一段（資金提供元 / 動物実験 / 出典が見つからないことの指摘）のいずれか
- PMID付きの参考文献が3件以上
- 本文2,000字以上
- 冒頭の HTML コメントにタイトル案3つ
- **研究を紹介するたびに「この試験に参加したのは誰か」を1行**（何人・何歳・どの国・どこの肌を測ったか）。
  `docs/style-guide.md` §4.7。**「n=48」とだけ書くのは、書いていないのと同じです。**
  読者は「この研究は私に当てはまるのか」を知りたいのに、その判断材料を渡されていません

**参考文献のタイトル・著者・年は、PubMed の表記と一字一句一致させること。** 後で機械が PubMed に問い合わせて突き合わせます。年は電子版の年ではなく、**掲載年**を書くこと。

薬機法に触れないこと。健康食品で効能を書かない。

### ★ すでに書いた記事に、話をつなぐこと

**記事どうしが繋がっていないと、読者は1本読んで帰ります。**

`site/articles.json` を読み、**すでに調べたテーマに関係する話が出てきたら、その語を本文で使ってください。**
（`linkTerms` に登録されている語です: ターンオーバー / 日焼け止め / SPF / レチノール /
ビタミンC誘導体 / 肌断食 / 飲むコラーゲン / 飲むヒアルロン酸 / PDRN など）

**その語を使えば、機械が自動でリンクを貼ります。** あなたがリンクを書く必要はありません。

```
○  この点は、レチノールの記事で見た「濃度を上げても見た目は変わらない」と同じ構図です。
   → 「レチノール」に自動でリンクが貼られる

✗  関連記事: [レチノールは濃いほうがいいのか](...)
   → リンクを手で書かない。機械が貼る
```

**★ ただし、関係が無いのに語を差し込まないこと。**
それは読者のためではなく、検索エンジンのための細工です。**うちが批判している側の手口です。**
（1記事につき最大4本まで、初出の1回だけ、機械が貼ります）

## 3.5 図版を1枚入れる（このブログの武器です）

**記事に、論文の数字をグラフにした図版を1枚入れてください。**

手順は2つです。

**① `site/figures.json` の `figures` に1つ追加する**

`_readme` に形式が書いてあります。既にある図版（`retinol-dropout` など）を見本にしてください。

- `type` は `bar`（横棒）/ `grouped`（2系列の横棒）/ `matrix`（2列の対比）/ `ladder`（順位はあるが数量ではないもの）から選ぶ
- **記事に書かれている数値だけを使うこと。** 図版のために新しい数字を作らない
- 値が確定していないもの（「観察期間内に戻らなかった」「その数字を報告した論文は0件」など）は `"pending": true` にする。実測値と同じ塗りにしないため

### ★ `sourcePmids` を必ず書く（これが無いと公開が止まります）

**図は、記事の中で最も強く記憶される部分です。** そこだけを検証の外に置かないでください。

```json
"sourcePmids": ["35778881"],
"sourceOther": null
```

- **`sourcePmids`** … その図の**数字が実際に載っている論文**の PMID（配列）
- **`sourceOther`** … PubMed に無い出典（日本語誌など）のときだけ。
  `{ "citation": "著者. タイトル. 誌名. 年;巻(号):頁.", "doi": "...", "why": "PubMed に収載されていないため" }`

**機械が照合します:**

1. `sourcePmids` に書いた PMID は、**その記事の参考文献にも載っていること**。
   **図だけが引いている論文があってはいけません。読者が図から原典に帰れなくなります**
2. `sourcePmids` が空なら `sourceOther` が必須。年が要ります

**★ 推測で PMID を埋めないこと。** それは来歴の捏造で、このブログが記事で追及していることそのものです。
特定できないなら、**その図を作らないでください。**

（`source` は、いまも読者向けの補足文として書けます。ただし**それは出典ではありません。**
「出典: 記事末尾の参考文献を参照」のような文で済ませないこと）

**② 記事の Markdown に1行入れる**

数字が出てきた直後に、この1行を置きます。

```
::figure:<キー名>::
```

**強い数字が無ければ、無理に作らないこと。** 図版のない記事があっても構いません。ただし、**数字が1つもない記事は、そもそもこのブログの記事ではありません。**

## 3.6 論文台帳に記入する（**これを忘れると公開が止まります**）

**このブログの資産は、記事ではありません。「論文の身元調査の記録」です。**

あなたは今日、論文を5本以上開き、`pubmed.mjs get` で**著者の所属・利益相反の申告・資金提供元**を見ました。**その記録を捨てないでください。**

`site/papers.json` の `papers` 配列に、**開いた論文を全部**追加します（記事に採用しなかった論文も、開いたなら書く）。

```json
{
  "pmid": "29949889",
  "title": "<PubMed のタイトルをそのまま>",
  "journal": "Nutrients",
  "year": 2018,
  "design": "ランダム化二重盲検プラセボ対照試験",
  "subjects": "女性64人・12週間",
  "fundingType": "industry",
  "fundingNote": "第1著者と第2著者の所属が Newtree（被験製品のメーカー）",
  "coiNote": "「利益相反なし」と申告している",
  "confirmed": "abstract",
  "articles": ["<記事のスラッグ>"]
}
```

**守ること:**

- **`fundingType` は5語のみ**: `industry` / `public` / `independent` / `none` / `unverified`
- **`confirmed` は3語のみ**: `fulltext` / `abstract` / `bibliographic`
- **効く・効かないは1文字も書かない。** 台帳は評価をしません。事実だけです（機械が検査します）
- **COI が「なし」と申告されていても、著者がメーカー社員なら `industry` です。** その食い違いを `coiNote` に書いてください。**これがこの台帳の武器です**
- **判断できないものは `unverified`。** 「たぶん企業資金だろう」で `industry` にしないこと
- 既に台帳にある論文を再び引用するときは、**`articles` にスラッグを足すだけ**（重複して追加しない）

**記事が引用した PMID が台帳に無ければ、機械が公開を止めます。** 調査したのに記録していないのは、このブログでは調査していないのと同じです。

## 4. サイトに登録する

`site/articles.json` の配列の**先頭に**1ブロック足す。

### date は「3日後」にすること

**今日の日付ではありません。** `date -d '+3 days' +%Y-%m-%d`（PowerShell なら
`(Get-Date).AddDays(3).ToString('yyyy-MM-dd')`）で取った日付を入れてください。

理由: **公開日が来るまでの3日間、この記事はメンバーのページにだけ出ます（先出し）。**
公開日が来たら、毎朝のビルドが自動で全員に出します。**あなたが後から何かをする必要はありません。**

- 「先出し用のフラグ」は作っていません。`build.mjs` は日付を見ているだけです
- 日付を今日にすると、先出しの期間がゼロになります。**メンバーが払う理由が1つ消えます**
- 日付を先にしすぎると、サイトの更新が止まって見えます。**3日です**

```json
{
  "slug": "<スラッグ>",
  "title": "<採用タイトル>",
  "subtitle": "<サブタイトル>",
  "date": "<YYYY-MM-DD>",
  "category": "<塗る / 飲む / 習慣 のいずれか>",
  "summary": "<トップページのカードに出る2〜3行>",
  "tags": ["<悩み>", "<悩み>"],
  "series": "<省略可。下の3つから>",
  "vol": "<series を書いたら必須。その連載の何回目か（数値）>",
  "linkTerms": ["<この記事を呼び出す語>"],
  "published": true
}
```

### linkTerms（他の記事から、この記事を呼び出す語）

**今後の記事の本文にこの語が出てきたら、機械が自動でこの記事にリンクします。**

- **その記事の主題を、はっきり指す語だけ**（例: `レチノール` / `肌断食` / `PDRN`）
- **一般的すぎる語を入れないこと。**「肌」「保湿」「成分」などは、あらゆる記事で誤爆します
- 1〜3語。多くしないこと

### series（連載）

単発の記事より、**連載の1回**のほうが読まれます。この記事が下のどれかに当てはまるなら、
`series` と `vol`（既存の最大値＋1）を入れてください。**当てはまらなければ、書かなくて構いません。**

| 連載名 | 中身 |
|---|---|
| `出典をたどる` | 世間の常識の出典を辿ったら、根拠が見つからなかった |
| `誰が資金を出したか` | 研究結果を分けていたのは、資金源だった |
| `数字の出どころ` | 売り場に並ぶ数字（28日・3分・SPF50）は、どこから来たのか |

- **この3つ以外を書くと verify.mjs が公開を止めます。** 新しい連載を作りたくなったら、
  勝手に足さず、報告に書いてオーナーの判断を仰ぐこと
- **無理にどれかへ入れないこと。** 全記事をどれかの連載に入れたら、それは分類であり「表」です
- **連載の一覧ページは作りません。** 作った瞬間、それは却下済みの「表」になります

### tags（読者の言葉で、記事への入口を作る）

カテゴリ（塗る／飲む／習慣）は**対象の分類**であって、読者の悩みではありません。
読者は「塗るもの」を探して来ません。**「毛穴」「乾燥」で来ます。**

**この記事が実際に扱った悩みだけ**を、次の語彙から**1〜3個**選んでください。

```
乾燥 / 毛穴 / シワ / シミ / ニキビ / 敏感肌 / 赤み /
ハリ / くすみ / 日焼け / たるみ / 角質 / テカリ
```

- **この語彙以外を書くと verify.mjs が公開を止めます。** 新しい語を足したくなったら、
  勝手に足さず、報告に「この語を足したい」と書いてオーナーの判断を仰ぐこと
- **「こちらの判定」をタグにしないこと。**「根拠なし」「動物実験」「メーカー資金」を
  タグにした瞬間、それは記事への入口ではなく**糾弾リストの目次**になります。却下済みの企画です
- 記事で扱っていない悩みを、集客のために足さないこと（それは釣りです）
- タグページは、記事が**2本以上**あるタグにだけ自動で作られます。1本しかないタグは
  記事側にリンクが出ないだけで、エラーにはなりません

## 5. 検証する（ここが関門）

```
node site/verify.mjs
```

- **通ったら** → 6へ
- **落ちたら** → エラーを読んで記事を直し、通るまで繰り返す。**3回直しても通らなければ、`site/articles.json` の published を false にして、何が通らなかったかを報告に書く。落ちた記事を無理やり公開しないこと。**

続けて `node site/build.mjs` を実行し、ビルドが通ることを確認する。

## 6. 公開する

- `topics.json` の該当項目の status を `"done"` にする
- `"queued"` が**5件未満**なら、新しいテーマを**5件**補充する。記事は平日毎日1本のペースで出るので、切らさないこと。基準は `topics.json` の `_readme` にある勝ちパターン（世間の常識の出典をたどると、根拠が無い／動物実験だった／メーカー資金だった）が再現できるもの
- **すでに `done` になっているテーマや、`articles/` に同じ内容の記事があるテーマを補充しないこと。** 重複した記事は、Google に「価値のない量産ページ」と判定される直接の原因になります
- `git add -A` → commit（メッセージは記事のテーマ）→ `git push origin main`

push すると GitHub Actions が自動でサイトを公開します。**Actions 側でも同じ verify.mjs が走るので、検証に落ちた記事は世に出ません。**

## 7. X の投稿を作って、予定表に積む

**これをやらないと、記事は増えるのに X の弾が尽きます。**

`.claude/agents/x-operator.md` と `docs/style-guide.md` を読み、**この記事から3本のポストを作って `x/queue.json` に追記**してください。

### 作るもの

| 種類 | 内容 |
|---|---|
| 単発ポスト × 2 | **リンクなし。** 論文の数字だけで完結するもの。1行目で指が止まること |
| 記事の告知 × 1 | **リンクあり。** 「更新しました」で終わらせず、記事で一番おいしい発見をその場で無料で渡す |

### 守ること

- **日時は、`x/queue.json` で最後に予約されている日の翌日以降**にすること。既存の予約とぶつけない
- **リンク付きは1日1本まで。** 既存の予約を見て、その日にリンク付きが既にあるなら別の日にする
- 時間帯は **朝8:00 / 昼12:30 / 夜21:00**（告知は夜）
- **日本語1文字＝2カウント、URLは23カウント固定、上限280。必ず数えること**
- 記事に書かれていない数値を書かない
- **既存の予約と内容が重複しないこと**（X の自動化ポリシーで重複投稿は禁止です）
- 煽らない（「知らないと損」「9割が間違えている」は禁止）
- `id` は一意に（例: `2026-07-30-golden-time-1`）

### 記事URL

`https://biyou-ronbun.com/articles/<スラッグ>.html`

## 8. 数字の画像を作る（強い数字があれば）

記事の中に、**1枚の画像にして人を止められる数字**があれば、`x/cards.json` の `cards` 配列に1つ追加してください。

- `type` は `bar`（横棒）/ `matrix`（2列の対比）/ `list`（数字の羅列）から選ぶ
- **記事に書かれている数値だけを使うこと。** 画像のために新しい数字を作らない
- `source`（出典）を必ず書く。**画像だけが独り歩きするので、出典が無ければ、このブログが批判している「出典のない数字」そのものになります**
- 追加したら、上で作った単発ポストのどれか1本に `"image": "x/cards/<スラッグ>.jpg"` を添える

画像そのものは、このあと `site/xcards.ps1` が自動で生成します（あなたは JSON を書くだけ）。

**強い数字が無ければ、無理に作らないこと。** 画像のない記事があっても構いません。

## 9. 最後に報告する

- 書いた記事のスラッグとタイトル
- **その記事の一番おいしい発見**（他のブログが書けない一段は何だったか）
- 検証で落ちた項目があれば、その内容
- 公開できたかどうか
- ネタを補充したなら、そのテーマ

**書けなかった場合や検証に通らなかった場合は、無理に公開せず、そのまま報告してください。**
